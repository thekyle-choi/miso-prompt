# B: 경계값 (Boundary)

최소/최대 입력이나 극단적인 요구사항으로 MISO의 처리를 확인하고, PRD까지 완주합니다.

## 테스트 케이스

| ID | 테스트 | 수행 방법 | 완료 조건 |
|----|-------|----------|----------|
| B-01 | 최소 입력 | 가능한 최소한의 정보로 진행 → MISO가 추가 질문으로 보완 | PRD 생성 완료 |
| B-02 | 최대 입력 | 모든 옵션 선택, 최대 정보 제공 → MISO 처리 확인 | PRD 생성 완료 |
| B-03 | 불가능 기능 | 구현 불가능한 기능만 요청 → MISO 대안 제시 → 수정하여 진행 | PRD 생성 완료 |

## B-01: 최소 입력

가능한 최소한의 정보만 제공하여 MISO가 적절히 보완하는지 확인합니다.

### 시나리오
```
Turn 1: 매우 간단한 주제 (예: "앱 만들기")
Turn 2: form 질문에 최소한의 선택만
  - multiselect에서 1개만 선택
  - 선택 사항은 모두 건너뛰기
Turn 3~N: MISO가 추가 질문으로 정보 보완 → PRD 완주
```

### 확인 사항
- 부족한 정보를 인식하는가?
- 적절한 추가 질문을 하는가?
- 최소 정보로도 PRD를 만들 수 있는가?

## B-02: 최대 입력

모든 옵션을 선택하고 최대한의 정보를 제공합니다.

### 시나리오
```
Turn 1: 매우 상세한 주제
  예: "GS25 점포의 재고관리, 유통기한 추적, 자동 발주,
       재고 실사, 폐기 처리를 통합 관리하는 앱"
Turn 2: form 질문에 모든 옵션 선택
  - multiselect에서 전부 선택
  - 선택 사항도 모두 입력
Turn 3~N: MISO가 과다 정보를 정리 → PRD 완주
```

### 확인 사항
- 많은 정보를 효과적으로 정리하는가?
- 우선순위를 적절히 판단하는가?
- 복잡도를 관리할 수 있는가?

## B-03: 불가능 기능

기술적으로 구현 불가능한 기능을 요청했을 때의 대응을 확인합니다.

### 시나리오
```
Turn 1: 불가능한 기능 요청
  예: "타임머신 재고관리 앱 (과거로 가서 재고 확인)"
Turn 2: MISO 불가능 판정 + 대안 제시
Turn 3: 사용자 대안 거부 또는 추가 불가능 요청
Turn 4: MISO 재설명 + 현실적 대안
Turn 5~N: 점진적 타협 → 실현 가능한 기능으로 수렴 → PRD 완주
```

### 확인 사항
- 불가능한 기능을 정확히 판단하는가?
- feasibility를 올바르게 표시하는가?
- 현실적인 대안을 제시하는가?
- 사용자를 설득할 수 있는가?

### 불가능 기능 예시
- 마음 읽기, 미래 예측, 시간여행
- 물리 법칙 위반 (순간이동, 무한 에너지)
- 개인정보/법률 위반 (타인 정보 무단 수집)
